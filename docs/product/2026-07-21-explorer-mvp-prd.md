# Explorer MVP PRD — Live Test Session

**状态：** 待评审（v0.1）

**日期：** 2026-07-21

**负责人：** Traceability 产品与工程团队
**关联模块：** `app/` Explorer、Agent Panel、`packages/core` replay、`server/`

---

## 1. 决策摘要

Explorer MVP 不做通用的多 Artifact 工作台，但会把“先打开网页、再决定是否录制或分析”做成零门槛的首要入口。

Explorer 是一个浏览与理解 Web 功能的工作台。用户先在浏览器式 Tab 中打开页面；当用户明确开始录制时，当前 Tab 才绑定为 **Live Test Session（实时测试会话）**。Traceability 随后将用户路径、页面状态、可控范围内的网络证据和人工标注关联成一条时间线；右侧 Agent 基于这些可追溯证据提出发现并生成可编辑的 Issue 草稿。

在该模型中：

- **Browser Tab / Browser Artifact** 是 MVP 唯一完整支持的 Artifact。它可以在没有录制的情况下独立存在；录制只是为当前 Tab 创建 TestSession 的显式操作。
- 操作、网络事务、截图/页面快照、圈选、Agent 发现和 Issue 草稿是此会话产生的**证据对象**，而不是需要独立导航的通用 Artifact 面板。
- MVP 使用一个 Chrome 式的“新建 Tab”作为浏览入口，但每个桌面窗口只允许 **一个可见 Browser Tab 和一个正在录制的测试会话**。已结束会话可回看；真实的多 Browser Tab、通用 Artifact Tab 和跨会话并行录制后置。

这项决策同时支持两条价值路径：**打开页面 → 圈选并理解功能**，以及 **操作 → 证据 → AI 分析 → 人工确认的 Issue 草稿**。

## 2. 背景与问题

当前 Traceability 已覆盖生产/预发场景下的异常采集、Issue 聚合和 rrweb 回放：

- `packages/core` 可以录制 rrweb 事件，但当前是 SDK 内的滚动缓冲，并主要在错误发生时上传。
- server 已能保存和查询回放；app 已能播放回放。
- Explorer 页面当前仍是占位页；当前 WebSocket 仅实时广播 Issue 变化，Agent 也没有读取实时测试证据的能力。

这使测试人员仍需要在浏览器、抓包工具、截图工具、缺陷系统和 AI 对话之间反复切换。问题不是“是否存在足够多的日志”，而是 AI 与测试人员都缺少同一条、可验证的测试证据链。

### 2.1 目标用户

**主要用户：功能探索者（产研、设计、测试与研发）**

- 产品、研发、设计、测试人员或技术负责人，需要理解一个页面、一个流程或一个局部区域“在做什么、为何如此工作”。
- 可以从应用已配置的入口进入，也可以直接粘贴一个待探索的网址。
- 希望把探索过程中的页面区域、用户行为和系统行为沉淀为可供 AI 和团队继续理解的证据。

**高频场景用户：质量与验证人员（QA / 测试 / 开发自测）**

- 在测试或预发环境中按业务流程操作 Web 应用。
- 需要迅速确认“刚才发生了什么、接口返回了什么、这是否是缺陷”。
- 希望将完整证据组织成缺陷，而不是手工复制请求、截图和复现步骤。

**协作消费者：研发与技术负责人**

- 需要从探索记录或 Issue 草稿中获得可复现路径、关联的网络上下文和页面证据。
- 需要判断 Agent 的结论是否有依据，而不是只接受黑盒描述。

### 2.2 用户任务（Jobs to Be Done）

1. 当我想理解一个页面或功能时，我希望直接输入网址并打开它，再圈选区域让 AI 解释，而无需先完成应用配置或开启录制。
2. 当我在测试中遇到可疑行为时，我希望录下我的操作和关键接口结果，以便无需重新复现就能解释问题。
3. 当我圈选某个页面区域时，我希望 AI 能结合该区域、当前路由、最近操作和接口证据说明它的业务作用或异常原因。
4. 当我确认是问题时，我希望一键得到可编辑的 Issue 草稿，并保留证据链接，而不是让 AI 自动对外创建 Issue。

## 3. 目标、非目标与成功标准

### 3.1 MVP 目标

1. 让任何功能探索者在 Explorer 中直接打开 HTTPS 网页；缺少应用入口配置不能阻塞浏览。
2. 让用户在打开的页面上圈选区域，并基于最小页面证据请求 AI 解释功能。
3. 让质量与验证人员在同一 Browser Tab 中显式启动一次受控 Web 测试录制。
4. 将操作、导航、网络元数据、有限且脱敏的 JSON 响应、页面标注统一到按时间排序的证据链。
5. 让右侧 Agent 面板能针对当前会话给出**带证据引用**的实时分析和 Issue 草稿。
6. 使人工确认前不会创建、更新或提交任何 Issue。

### 3.2 明确非目标

- 不支持任意网页的“无约束完整抓包”。
- 不默认保存密码、Cookie、认证头、上传文件、原始表单内容或无限制响应体。
- 不支持多个正在录制的浏览器 Tab，也不支持通用的多类型 Artifact 框架。
- 不提供浏览器自动化、脚本回放、接口改写、流量 mock 或自动修复。
- 不自动向 Jira、飞书、GitHub 或其他外部系统创建 Issue。
- 不将 AI 对页面或接口内容的解释视为事实；每个发现必须可回到证据。

### 3.3 Pilot 成功标准

在一个已批准的测试/预发应用中，测试人员可以独立完成以下闭环：

1. 开始录制并走完一个业务路径。
2. 在时间线中找到操作、对应接口结果和一个页面标注。
3. 获得一条引用这些证据的 Agent 发现。
4. 生成并编辑 Issue 草稿。
5. 研发可从草稿中理解复现路径和关键证据，而无需测试人员再口头补充。

Pilot 期间需要持续观察：会话完成率、有效 Issue 草稿转化率、Agent 发现被采纳率、因隐私或无关流量被过滤的事件比例，以及用户完成上述闭环所需时间。

## 4. 产品范围

### 4.1 MVP 范围内

| 能力 | MVP 行为 |
| --- | --- |
| 浏览器式首个 Tab | 首次进入始终展示一个 `新建 Tab`：应用有配置入口时显示为推荐快捷项；没有配置入口时地址栏自动聚焦，用户可直接输入 URL 并按 Enter 打开。 |
| 受控浏览器 | 在 Explorer 主区打开目标页面，支持后退、前进、刷新、地址变更与加载状态。浏览网页本身不自动创建录制数据。 |
| 会话创建 | 用户在已打开页面上点击 `开始录制`，选择/确认当前 Traceability 应用和可选测试标题；系统才为当前 Tab 创建 TestSession。 |
| 页面理解 | 未录制状态也支持“选择元素”和“分析此处”；仅生成一次性的截图/元素语义/当前 URL 证据，不采集连续操作或网络响应。 |
| 录制控制 | 显式开始、暂停、继续、停止；录制状态始终可见。 |
| 行为证据 | 记录导航、点击、表单提交、元素选择、输入事件的脱敏摘要，以及 rrweb 回放数据。 |
| 网络证据 | 记录 fetch/XHR 的 URL、方法、状态码、耗时、内容类型、体积和关联时间；对获准的 JSON 接口保存脱敏且有大小上限的响应摘要。 |
| 证据时间线 | 以单一时序展示操作、网络、导航、标注、错误和 Agent 发现；支持按类型筛选与点击定位。 |
| 页面标注 | “选择元素”模式；保存选中元素的截图、坐标、DOM/无障碍语义摘要、当前 URL 和用户备注。 |
| Agent 分析 | 当前 Agent 会话绑定 Explorer 会话；在用户请求、标注、失败网络请求和会话结束等节点分析证据。 |
| Issue 草稿 | 根据人工选择的证据生成可编辑草稿；包含标题、描述、复现步骤、影响、证据引用和 Agent 假设。 |
| 会话回看 | 停止后的会话保留摘要、时间线和可用 replay，供当前用户回看。 |

### 4.2 后置范围

| 后置能力 | 后置原因 |
| --- | --- |
| 多个活跃 Browser Tab / 会话 Tab | 首个 Tab 足以验证“直接打开—理解—录制”的核心流程；原生浏览器视图的生命周期、焦点、登录态与资源开销在多 Tab 下显著增加。 |
| 任意 Artifact 类型及可插件化布局 | 目前没有第二类 Artifact 的稳定用户任务；过早抽象会掩盖会话数据模型。 |
| 自由手绘圈画 | 首版“元素选择 + 截图标注”已能提供更强的 DOM 语义和更可靠的 AI 上下文。 |
| 跨会话相似问题聚类 | 需要足够的会话样本和可靠的证据标准化。 |
| 自动创建外部 Issue / 自动修复 | 需要权限模型、审批、外部系统集成和更强的安全边界。 |
| 浏览器扩展、移动端设备模拟、网络改写 | 属于测试自动化/调试工具范畴，超出本 MVP。 |

## 5. 核心用户流程

### 流程 A：首次打开并探索一个网页

1. 用户进入 Explorer，看到一个类似 Chrome 的 `新建 Tab`，而不是配置表单或空白 Artifact 工作台。
2. 如果当前 Application 配置了推荐入口，页面显示可点击的入口快捷项；用户仍可编辑地址栏并打开任意 HTTPS URL。
3. 如果当前 Application 没有配置入口，地址栏自动聚焦，用户粘贴/输入 URL 后按 Enter 即可打开页面。
4. 页面打开后处于“浏览与理解”模式：录制默认关闭，工具条提供 `选择元素`、`分析当前页面` 与 `开始录制`。
5. 用户可选择一个页面区域并询问 Agent；系统只生成本次分析所必需的标注证据。

### 流程 B：从已打开页面开始一段测试录制

1. 用户在当前 Browser Tab 中点击 `开始录制`。
2. 系统确认应用归属、当前域名与本次采集范围：操作摘要、网络元数据、响应捕获规则、数据保留说明。
3. 用户显式确认后，当前 Tab 绑定 TestSession；顶部常驻高辨识度的录制状态、时长和暂停/停止按钮。
4. 用户正常操作页面。系统在不阻塞网页的前提下追加证据到时间线。

### 流程 C：在异常点向 AI 询问

1. 用户在页面中开启“选择元素”，点击可疑区域，或在时间线选择某个失败请求。
2. 系统创建一条标注/网络证据，并显示其与最近操作、当前路由的关联。
3. 用户点击“分析此处”或在 Agent 面板直接提问。
4. Agent 读取经过脱敏和裁剪的会话证据，输出：观察到的事实、可能原因、置信度、尚缺信息和证据引用。
5. 用户可以追溯到对应时间线事件、截图或 replay；Agent 不可把未经验证的推测表述为事实。

### 流程 D：生成 Issue 草稿

1. 用户在 Agent 发现或时间线中选择“生成 Issue 草稿”。
2. 系统预填标题、影响描述、复现步骤、环境、相关网络证据、截图/标注和 Agent 分析。
3. 用户可编辑、移除任意证据或补充说明。
4. 用户显式确认“创建草稿”。MVP 仅在 Traceability 内创建草稿，绝不自动发送至外部系统。
5. 用户停止会话；系统完成持久化并显示会话摘要。

## 6. 信息架构与界面要求

Explorer 沿用现有应用 Shell：左侧为主工作区，右侧复用 Agent Panel。它不另起一套 Artifact 管理导航。

### 6.1 首次入口 / 新建 Tab

- 顶部以浏览器式 Tab 承载 `新建 Tab`，下方是后退、前进、刷新与地址栏；地址栏接收完整 URL，输入不带协议时默认补全为 `https://`。
- 有当前 Application 推荐入口时，显示“打开 <入口名称>”快捷项；无推荐入口时不显示配置阻塞，直接将焦点放在地址栏。
- 用户按 Enter 后立即加载页面。此时仅是浏览，不自动开启 rrweb、操作录制、网络采集或 Agent 持续分析。
- 页面加载成功后显示 `选择元素`、`分析当前页面` 和主按钮 `开始录制`。无当前 Application 归属时，用户仍可浏览和单次分析；开始录制或创建 Issue 草稿前必须选择应用归属。

### 6.2 浏览与理解模式

- Browser Artifact 占据主区域，地址栏保留可编辑状态。
- `选择元素` 进入 inspect 模式：用户下一次点击建立标注，避免触发网页自身操作；Esc 退出。
- `分析当前页面` 提交当前 URL、可见页面快照和最小 DOM/无障碍语义摘要给 Agent，不读取历史网络流量。
- 页面未开始录制时，时间线不显示连续行为或网络事件；界面明确提示“浏览中，未录制”。

### 6.3 进行中的会话

- 顶部控制条：后退、前进、刷新、地址栏、页面加载状态、录制时长、暂停/停止。
- 主区域：Browser Artifact。
- 可折叠底部证据时间线：按时间展示操作、导航、网络、标注、错误和 Agent 发现；支持“全部 / 操作 / 网络 / 标注 / 异常”过滤。
- 工具：`选择元素`、`添加备注`、`分析当前步骤`、`生成 Issue 草稿`。
- 右侧 Agent Panel 明确显示“正在分析：<测试会话标题>”，避免用户误以为 Agent 在阅读全部历史会话。

### 6.4 已停止会话

- 顶部显示会话状态、时长、证据计数、是否已有草稿。
- Browser 区可切换至 replay；时间线和 Agent 发现仍可浏览。
- 支持 `重新开始新的测试会话`，但不支持从旧会话恢复录制。

### 6.5 状态与错误

- URL 无效或协议不受支持：阻止打开并说明原因。未获录制授权的 HTTPS 域名仍可浏览，但不可自动开启采集；用户必须在开始录制时确认或取得授权。
- 页面加载失败：记录失败导航，但保持会话可停止/重试。
- 录制或网络采集失败：显示降级提示；网页仍可继续使用，不能静默假装采集成功。
- 响应因隐私、大小或 MIME 类型被拒绝：时间线显示“已过滤”的原因，不显示原始内容。
- Agent 不可用：证据仍持续录制；用户可稍后基于完整会话请求分析。

## 7. 证据与数据定义

录制期间的证据使用稳定的 `testSessionId` 关联；未录制的页面理解操作仅以 `browserTabId` 与一次性 `explorationId` 关联。以下是产品契约，不限定最终 TypeScript 名称或存储实现。

| 对象 | 核心字段 | 说明 |
| --- | --- | --- |
| `BrowserArtifact` | tabId、当前 URL、导航历史摘要、浏览器分区标识、可选 sessionId | Explorer 的首个可见 Tab；可在没有 TestSession 时存在。 |
| `TestSession` | id、appId、browserTabId、title、targetOrigin、状态、开始/结束时间、capturePolicy | 用户显式开始录制后创建的主聚合对象。 |
| `EvidenceEvent` | id、browserTabId、可选 sessionId、timestamp、type、summary、privacyStatus、关联 evidenceIds | 时间线最小单元。 |
| `InteractionEvidence` | action、元素角色/名称/选择器摘要、坐标、输入是否已掩码 | 用户操作的可读摘要；不保存敏感文本。 |
| `NetworkEvidence` | requestId、URL 摘要、method、status、duration、mimeType、响应摘要/过滤原因 | 只捕获在采集策略允许范围内的信息。 |
| `AnnotationEvidence` | screenshotRef、boundingRect、元素语义、route、用户备注 | “圈选/指定区域”的可追溯表达。 |
| `AgentFinding` | id、sessionId、claim、confidence、evidenceIds、unknowns、状态 | 必须与证据关联；含“待验证”状态。 |
| `IssueDraft` | id、sessionId、title、description、steps、evidenceIds、状态 | 用户可编辑；只有确认后才创建。 |

### 7.1 数据采集规则

1. **操作记录**：记录操作类别与目标元素的语义摘要；输入内容默认掩码，仅保存“已输入/长度区间/字段类型”等安全信息。
2. **网络记录**：默认仅记录元数据。响应体仅在同时满足以下条件时保存摘要：请求属于允许的域名/路径规则、`Content-Type` 为 JSON、完成敏感字段脱敏、大小不超过试点配置上限。
3. **rrweb 回放**：用于回看页面状态与操作路径；遵从 `maskAllInputs`，并在页面/元素规则要求时屏蔽区域。
4. **证据关联**：操作与网络事件以时间窗口和当前路由关联；系统可以给出关联候选，但不能把候选关系显示为确定因果。
5. **AI 输入**：Agent 只获得已脱敏、限量、带来源 ID 的证据切片。原始页面、接口返回和用户备注均视为不可信内容，不可作为 Agent 指令。

## 8. Agent 行为要求

### 8.1 触发时机

Agent 分析仅在以下场景运行：

- 用户在浏览与理解模式中选择页面/区域后主动提问，或在当前录制会话内主动提问。
- 用户标注元素后选择“分析此处”。
- 允许范围内的请求出现 4xx/5xx、解析失败或显著异常模式。
- 用户停止会话并请求总结。

不允许对每一次点击或每个响应单独发起模型调用，也不允许将原始响应持续灌入对话历史。

### 8.2 输出格式

每条 Agent 发现至少包含：

- **观察到的事实**：只陈述证据直接支持的内容。
- **可能原因**：明确是推测，并给出置信度。
- **证据**：可点击的时间线事件/标注/回放引用。
- **未确认信息**：建议测试人员补充的验证动作。
- **下一步**：如“比较成功与失败请求”“检查某字段是否允许为空”。

### 8.3 权限边界

- Agent 只读 Explorer 证据；不能操作被测网页、修改请求、创建外部 Issue 或提交修复。
- Agent 可以生成 Traceability 内的 Issue 草稿，但“创建草稿”仍需用户点击确认。
- 页面文本、接口 payload、用户备注和第三方内容都可能包含提示注入；它们只能被当作数据解释，不能修改系统指令或授权范围。

## 9. 隐私、安全与合规要求

这是 MVP 的发布门槛，而非优化项。

1. **浏览与录制分离**：用户可在新建 Tab 输入并浏览 HTTPS URL；这不会自动采集页面或网络数据。开始录制、持续 AI 分析或创建 Issue 前，必须确认当前 Application 与目标域名的采集授权。本地开发可单独允许 loopback HTTP。
2. **独立浏览器上下文**：Browser Artifact 使用隔离的 Electron session/profile，不复用 Traceability 桌面应用的 Cookie 或凭据。
3. **远程内容隔离**：远程页面禁用 Node integration，开启 context isolation 与 sandbox；限制权限请求、新窗口和非批准导航。
4. **默认脱敏**：永不采集 Authorization、Cookie、Set-Cookie、密码字段、文件内容；响应 JSON 按键名规则脱敏，并允许应用配置额外规则。
5. **最小化存储**：进行中的会话只保留 Agent 所需的滚动证据窗口；停止后持久化已选择/已关联的证据。试点的默认保留期建议为本地 7 天，服务端只保存用户附到 Issue 的证据，待评审确认。
6. **可见性与可撤销性**：用户始终看得到正在录制什么；可停止录制、删除未提交会话，并在时间线看到每个响应为何被过滤。

Electron 对远程内容明确要求禁用 Node integration、开启 context isolation 与 sandbox，并配置权限和导航控制；MVP 必须满足该基线。参考：[Electron Security](https://www.electronjs.org/docs/latest/tutorial/security)。

## 10. 技术边界与实现假设

本节用于约束实现方向，不是详细技术设计。

1. **网页承载**：采用 Electron `WebContentsView` 作为 Browser Artifact。当前 Electron 版本不应以已不推荐的 `<webview>` 或已废弃的 `BrowserView` 为实现基础。参考：[WebContentsView](https://www.electronjs.org/docs/latest/api/web-contents-view)、[webview warning](https://www.electronjs.org/docs/latest/api/webview-tag)。
2. **原生视图布局**：`WebContentsView` 是原生子视图，不能像 React DOM 一样自然嵌套。因此需要由 main process 管理其生命周期和边界，并由 Explorer 页面通过受控 IPC 上报可视区域；这是首个工程验证点。
3. **行为采集**：Explorer 需要专用的 guest preload/capture service。现有 core rrweb 实现可复用事件格式与脱敏原则，但其“SDK 内、错误时上传”的生命周期不能直接满足 Live Test Session。
4. **网络采集**：Electron `webRequest` 负责网络生命周期元数据。若需要响应体，技术验证需确认使用 Chrome DevTools Protocol 的 `Network.getResponseBody` 在目标 Electron 版本、重定向、Service Worker 和导航场景下的稳定性；不可用时降级为元数据和页面侧 fetch/XHR 摘要。参考：[Electron Debugger](https://www.electronjs.org/docs/latest/api/debugger/)、[CDP Network](https://chromedevtools.github.io/devtools-protocol/tot/Network/)。
5. **实时分发**：服务端需要从仅广播 Issue 变化扩展为会话级实时事件；Agent 通过一个只读的、按游标查询证据的工具读取会话，而不是通过无限增长的 prompt 接收原始事件。
6. **持久化**：原始高频事件不能无分页地堆进当前 replay 记录。实现需要分离“会话索引、分块证据、已选附件和 Agent 发现”，并支持删除和保留策略。

## 11. MVP 验收标准

### 11.1 功能验收

- [ ] 当前 Application 没有配置入口时，用户首次进入 Explorer 能看到自动聚焦的 `新建 Tab` 地址栏，输入 URL 并按 Enter 后可打开页面。
- [ ] 当前 Application 配置了入口时，用户可一键打开推荐入口，也可编辑地址栏访问其他 HTTPS URL。
- [ ] 用户在未录制状态可选择页面元素并请求一次性 Agent 分析；连续操作、网络响应与 rrweb 不会被采集。
- [ ] 用户可在已打开的 Browser Tab 中显式开始一段测试会话。
- [ ] 用户可在 Browser Artifact 中完成导航和登录；录制状态、时长、暂停和停止始终清晰可见。
- [ ] 一条典型路径能按时间展示“点击/提交 → XHR 或 fetch → 状态码/脱敏响应摘要 → 页面后续状态”。
- [ ] 用户可选中一个页面元素，留下带截图、元素语义和备注的标注，并要求 Agent 分析。
- [ ] Agent 的每条发现均关联至少一个证据 ID，且将假设与事实分开表述。
- [ ] 用户可基于会话生成、编辑并确认 Traceability 内的 Issue 草稿。
- [ ] 停止会话后可回看时间线与可用 replay。

### 11.2 安全与可靠性验收

- [ ] 被测远程页面无法使用 Node/Electron API，也无法任意调用 Traceability 主进程 IPC。
- [ ] 未经显式允许的域名不会自动开启录制或采集；弹窗和权限请求不会自动放行。
- [ ] 密码、Cookie、Authorization、文件内容和被脱敏字段不会出现在时间线、Agent 上下文、日志或 Issue 草稿中。
- [ ] 录制、网络采集或 Agent 服务失败时，被测网页仍可用，且 Explorer 清楚显示降级状态。
- [ ] 用户删除未提交会话后，相关本地证据不再可由 Explorer 或 Agent 查询。

## 12. 分阶段交付与发布门槛

### Phase 0：技术验证（发布前门槛）

使用一个受控的测试页面证明以下链路：

`页面操作 → 网络事件 → 已脱敏响应摘要 → 页面标注 → Agent 证据引用`

必须同时验证：`WebContentsView` 的布局与焦点、登录流程、重定向、响应体捕获降级、敏感字段脱敏、关闭会话后的资源释放。

技术验证失败时，不进入 UI 铺开开发；先选择“仅 SDK 受控应用采集”或“仅网络元数据”作为降级方案。

### Phase 1：Explorer MVP（内部 Pilot）

交付本 PRD 第 4.1 节的范围，限定一个批准的测试/预发应用、受控用户和明确的数据保留策略。

### Phase 2：基于 Pilot 数据的后续决策

仅当 Pilot 证明多会话切换是高频需求时，添加“会话 Tab”（每个 Tab 仍是 TestSession）；仅当第二种 Artifact 有独立任务、独立生命周期和稳定用户价值时，才抽象 Artifact 工作台。

## 13. 评审时需要确认的决策

1. 是否同意“浏览任意 HTTPS URL 不采集；开始录制/持续分析时确认 Application 与域名授权”的两阶段模型？建议同意，以保证零门槛入口与采集安全可以共存。
2. MVP 的首个 Tab 是否保持单 Tab，待 Pilot 验证后再加 `+` 和多 Tab 管理？建议保持单 Tab；Chrome 式入口不等于先做完整浏览器。
3. Pilot 的响应采集策略：仅网络元数据，还是允许按接口白名单保存脱敏 JSON 摘要？建议采用白名单 JSON 摘要。
4. 试点的本地会话默认保留期是否接受 7 天？服务端是否只保存附到 Issue 的证据？
5. “创建 Issue 草稿”在 MVP 中是否只创建 Traceability 内部草稿？建议是；外部 Issue 系统集成后置。
6. Agent 是否仅在用户显式请求时分析，还是允许在失败请求和结束会话时给出被动提示？建议允许后者，但不自动创建 Issue。

## 14. 与现有项目的关系

- Explorer 现有占位路由保留为入口，但会从静态页面升级为会话工作台。
- 现有 rrweb replay、server replay 记录与播放器是有价值的复用基础；Explorer 会引入不同于“错误触发上传”的主动会话生命周期。
- 现有 Agent Panel 是分析呈现层；MVP 不新建第二个 Agent，而是让当前 Agent 会话带有明确的 `testSessionId` 证据范围。
- 现有 server WebSocket 和数据模型需要扩展以承载会话级事件、证据查询与草稿，但不改变生产 Issue 监控的既有语义。
